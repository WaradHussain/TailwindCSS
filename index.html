Chapter 00

History & Evolution

Tailwind CSS didn't start as a framework. It started as one developer's frustration with existing tools, written during a live stream in 2017. Understanding its origin tells you everything about why it works the way it does.

The Origin Story

Adam Wathan was building a SaaS product called Nitpick CI in 2017. He was unhappy with Bootstrap's opinionated components and kept finding himself overriding styles. He started extracting the patterns he used repeatedly into small, single-purpose CSS utility classes. During a live stream, he open-sourced the result as a PostCSS plugin.

The founding team: Adam Wathan (creator, API design), Jonathan Reinink (early contributor), David Hemphill (early contributor), and Steve Schoger (design — he later co-authored Refactoring UI). Together they formed Tailwind Labs.

Complete Version Timeline

Adam live-streams the creation of a PostCSS utility plugin. Ships it publicly. ~50KB of utility classes. Minimal community attention.
v1.0 — May 13, 2019
Stable public release. PurgeCSS integration arrives for production size reduction. Establishes the spacing scale, color palette, and breakpoint system that remains today.
v1.4–v1.9 — 2019–2020
Grid utilities added. Transition and animation utilities ship. Ring utilities arrive. Accessibility improvements throughout.
v2.0 — November 2020
Dark mode support added. Extended color palette (220+ colors with all intermediate shades). Ring utilities for accessible focus rings. JIT compiler begins development behind the scenes.
v3.0 — December 2021
Game-changer: Just-In-Time (JIT) engine becomes the default. Arbitrary values with square brackets. Every color in the default palette. Play CDN for zero-setup prototyping. Build times drop from seconds to milliseconds.
v3.2 — October 2022
Container queries (@container) support arrives. CSS Cascade Layers (@layer) integration. has-[] variant for parent-state targeting.
v3.3–v3.4 — 2023
Extended P3 wide-gamut color support. line-clamp utilities promoted to core. Logical properties (ms-, me-, ps-, pe-). text-balance and text-pretty wrapping utilities.

v4.3.1 — June 12, 2026 (current)
First-party scrollbar styling utilities. New zoom and tab-size utilities. Better @variant support. This is the version you should be using today.

Chapter 01

Why Tailwind? The Problem It Solves

Before learning any tool, you must understand the pain it's designed to solve. Tailwind fixes a real problem with how we've been writing CSS.

The Old Way — Traditional CSS

In traditional CSS, you write HTML, give elements class names, then switch to a separate CSS file to write styles for those class names. This works, but creates real problems as your project grows:

❌ Traditional CSS — 2 Files
<!-- index.html -->
<div class="card">
  <h2 class="card-title">Hello</h2>
  <p class="card-text">World</p>
</div>
/* styles.css */
.card {
  padding: 16px;
  border-radius: 8px;
  background: white;
}
.card-title {
  font-size: 20px;
  font-weight: 700;
}
.card-text {
  color: #6b7280;
}
✅ Tailwind CSS — 1 File
<!-- index.html -->
<div class="p-4 rounded-lg bg-white">
  <h2 class="text-xl font-bold">
    Hello
  </h2>
  <p class="text-gray-500">
    World
  </p>
</div>

<!-- No CSS file needed! -->

Chapter 02

How Tailwind Works Under the Hood

Understanding what Tailwind actually does will save you hours of confusion. It's simpler than most people think.

Utility Classes — One Class, One Rule

Every Tailwind class maps to exactly one CSS rule. No more, no less.

/* These Tailwind classes produce this CSS: */
flex         →  display: flex;
text-center  →  text-align: center;
font-bold    →  font-weight: 700;
p-4          →  padding: 1rem;  /* 16px */
text-blue-500 → color: #3b82f6;
rounded-lg  →  border-radius: 0.5rem;
hidden       →  display: none;

The JIT Compiler — Tailwind's Superpower

JIT stands for Just-In-Time. Tailwind scans your HTML/JSX files, finds every class you've used, and generates only the CSS for those classes. Nothing more.

Without JIT vs With JIT

Important: Because JIT scans your files for class names, you must write complete class names. You cannot do string concatenation like "bg-" + color — Tailwind won't detect it. Always write the full class name: "bg-blue-500".

Is Tailwind Just Inline Styles?

This is the #1 misconception. The answer is no — and here's why it matters:

Media queries❌ Impossible✅ md:text-xl

Pseudo-elements❌ Impossible✅ before:content-['★']

Tailwind's CSS Layer System

Tailwind organizes its styles into three layers, in order of priority (lowest to highest):

LayerWhat it containsPriority
baseGlobal resets, default element styles (removes browser defaults)Lowest
componentsReusable component classes you createMedium
utilitiesAll the individual utility classes (p-4, flex, etc.)Highest

This means utility classes always win. You can add a component class .btn with some styles, then override any of them by adding a utility class directly in your HTML — the utility will win.

Chapter 03

JIT Engine — Deep Dive

The Just-In-Time engine is the most important technical innovation in Tailwind's history. Understanding how it works explains every "rule" you'll encounter — including why you can't use dynamic class names.

How the JIT Pipeline Works

Every time you save a file, Tailwind's engine runs a 4-stage pipeline in milliseconds:

1. Scan Files
Reads every .html, .jsx, .tsx, .vue file matching your content config
2. Detect Classes
Regex tokeniser extracts every candidate string that could be a Tailwind class
3. Generate CSS
For each valid class found, generates the exact CSS rule on-demand
4. Output Bundle
Only the classes actually used — nothing more. 5–20KB typical output

The #1 Rule — No Dynamic Class Names

Because Tailwind scans source files as plain text (not executing JavaScript), it can only detect class names that exist as complete strings. The engine cannot know what a runtime expression will evaluate to.

❌ BROKEN — class name split across JS
// Tailwind CANNOT detect this:
const color = 'blue';
<div className={`bg-${color}-500`}>
  Never gets the bg-blue-500 CSS
</div>
✅ CORRECT — complete class names only
// Map complete strings — Tailwind CAN detect:
const colors = {
  blue: 'bg-blue-500',
  red:  'bg-red-500',
};
<div className={colors[color]}>
  Works perfectly
</div>
Safelist for truly dynamic classes: If you receive class names from a CMS, API, or user input that Tailwind can't scan, add them to your safelist in the config. They'll always be included in the output regardless of whether the scanner finds them.

v4 Engine — Rust & Lightning CSS

Tailwind v4 completely rewrote the engine in Rust via Lightning CSS, replacing the Node.js/PostCSS pipeline. This wasn't just about speed — it enabled new capabilities:

Incremental rebuild~100ms~4ms

Arbitrary Values — How the Scanner Handles Them

When Tailwind sees bg-[#1a2030], the JIT engine's tokeniser extracts the bracket content and generates a unique CSS class on-demand. This is why arbitrary values work without any configuration:

/* What you write */
.bg-\[#1a2030\] {
  background-color: #1a2030;
}

/* Arbitrary with calc */
.top-\[calc\(100vh-4rem\)\] {
  top: calc(100vh - 4rem);
}

/* Even CSS variables as values */
.text-\[--brand-color\] {
  color: var(--brand-color);
}
v4 shorthand for CSS variables: In v4, you can write text-(--brand-color) instead of text-[var(--brand-color)]. Parentheses are the new shorthand for CSS variable values.

# 1. Create Vite project
npm create vite@latest my-project

# 2. Install Tailwind and Vite plugin
npm install tailwindcss @tailwindcss/vite

# 3. In vite.config.js / vite.config.ts
import tailwindcss from '@tailwindcss/vite'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [
    tailwindcss(),
  ],

/* In your main CSS file (e.g. src/style.css) */

The Base Reset — Tailwind's Starting Point

Tailwind automatically removes all browser default styles. This surprises beginners but is one of its biggest advantages.

What Gets Removed

ElementBrowser Default (removed by Tailwind)You Add Back
h1–h6Bold, large font sizes, marginstext-3xl font-bold mb-4

Why is this good? Browser defaults are inconsistent across Chrome, Firefox, and Safari. Tailwind removes them all so you start from a clean slate that looks the same everywhere. You're not fighting browser styles — you're building on a blank canvas.

Re-adding Defaults in Your CSS

If you want certain elements to always be styled (e.g., all H tags bold by default), add them to the base layer in your CSS:

@import "tailwindcss";

@layer base {
  h1, h2, h3, h4, h5, h6 {
    font-weight: 700;
  }
  a {
    color: inherit;
    text-decoration: none;
  }
}

Space Between Children

Instead of adding margin to every child, add space-y-{n} or space-x-{n} to the parent. It adds margin-bottom to all children except the last.

<!-- Without space-y: must add mb to each child -->
<div>
  <p class="mb-4">First</p>
  <p class="mb-4">Second</p>
  <p>Third</p>
</div>

<!-- With space-y: cleaner, same result -->
<div class="space-y-4">
  <p>First</p>
  <p>Second</p>
  <p>Third</p>
</div>
space-y vs gap: Use space-y for regular block elements stacked vertically. Use gap when you're using flexbox or grid — gap works better in those contexts.

Flexbox

Flexbox is for one-dimensional layouts — arranging items in a row or column. Add flex to the parent, then use flex classes to control alignment and spacing.

Dark Mode

Tailwind supports dark mode natively. Add dark: before any class and it only applies when dark mode is active.

Mode 1 — System Preference (Automatic)

By default, Tailwind uses the user's OS/browser dark mode setting. No setup needed:

<div class="bg-white text-gray-900 dark:bg-gray-900 dark:text-white">
  This adapts to your system dark mode setting.
</div>

Mode 2 — Manual Toggle (Custom Class)

To give users a toggle button, configure Tailwind to use a class:

/* In your CSS file */
@import "tailwindcss";

/* Tell Tailwind: activate dark: when .dark class is on html/body */
@custom-variant dark (&.dark, .dark &);
// Toggle dark mode with JavaScript:
const toggle = () => {
  document.documentElement.classList.toggle('dark')
}

// Or: save preference to localStorage
const savedTheme = localStorage.getItem('theme')
if (savedTheme === 'dark') {
  document.documentElement.classList.add('dark')
}

  Page content
</div>

Use @layer base to set styles that apply globally to HTML elements. Good for setting defaults you always want:

@layer base {
  /* All headings bold by default */
  h1, h2, h3, h4, h5, h6 { font-weight: 700; }

  body { scrollbar-color: #374151 #111827; }

Configuration — Design Tokens & Customisation

Tailwind's configuration is how you turn a generic utility framework into your product's design system. v3 and v4 take completely different approaches — both worth knowing.

v3 — tailwind.config.js

In v3, all configuration lives in a JavaScript file at the root of your project. This gives you the full power of JS — loops, conditionals, imported values from other files.

/** @type {import('tailwindcss').Config} */
module.exports = {
  // Where to scan for class names
  content: [
    './src/**/*.{html,js,jsx,ts,tsx,vue}',
    './public/index.html',
  ],

  // Dark mode strategy
  darkMode: 'class', // or 'media'

  theme: {
    // EXTEND — add to the defaults (recommended)
    extend: {
      colors: {
        brand: {
          50:  '#eff6ff',
          500: '#3b82f6',
          900: '#1e3a8a',
        },
        surface: '#0d1117',
      },
      fontFamily: {
        display: ['Cabinet Grotesk', 'sans-serif'],
      },
      spacing: {
        '18': '4.5rem',  // 72px
        '128': '32rem', // 512px
      },
      borderRadius: {
        '4xl': '2rem',
      },
      animation: {
        'fade-in': 'fadeIn 0.5s ease-in-out',
      },
      keyframes: {
        fadeIn: {
          '0%': { opacity: '0' },
          '100%': { opacity: '1' },
        },
      },
    },

    // REPLACE (no extend) — completely replaces defaults
    // screens: { sm: '640px', md: '768px' }, ← replaces ALL breakpoints
  },

  plugins: [
    require('@tailwindcss/typography'),
    require('@tailwindcss/forms'),
  ],
};
extend vs top-level: Adding under theme.extend.colors ADDS to the palette. Adding directly under theme.colors REPLACES the entire palette — you lose all the built-in colors. Always use extend unless you intentionally want to start from scratch.

v4 — @theme in CSS

In v4, configuration moves entirely into your CSS file. No JavaScript, no separate config file. Design tokens become CSS custom properties that are available everywhere — in Tailwind classes, in your own CSS, and in JavaScript via getComputedStyle().

@import "tailwindcss";

@theme {
  /* Colors: prefix with --color- */
  --color-brand-50:  oklch(97% 0.02 250);
  --color-brand-500: oklch(60% 0.18 250);
  --color-brand-900: oklch(25% 0.12 250);
  --color-surface:   #0d1117;

  /* Fonts: prefix with --font- */
  --font-display: 'Cabinet Grotesk', sans-serif;

  /* Spacing: prefix with --spacing- */
  --spacing-18: 4.5rem;
  --spacing-128: 32rem;

  /* Breakpoints: prefix with --breakpoint- */
  --breakpoint-xs: 475px;
  --breakpoint-3xl: 1920px;

  /* Animations */
  --animate-fade-in: fadeIn 0.5s ease-in-out;
}

@keyframes fadeIn {
  from { opacity: 0; }
  to   { opacity: 1; }
}

These tokens are now usable as native CSS variables anywhere — in Tailwind classes like bg-brand-500, in your CSS as var(--color-brand-500), and even in JavaScript:

// Access design tokens in JavaScript (v4)
const brand = getComputedStyle(document.documentElement)
  .getPropertyValue('--color-brand-500')
  .trim(); // "oklch(60% 0.18 250)"

The cn() Pattern — The Essential Helper

Every serious Tailwind project uses cn(). It solves a real problem: when you're conditionally applying classes, Tailwind classes can conflict and the later class doesn't always win. cn() merges classes intelligently.

Install two packages:

npm install clsx tailwind-merge

Create the utility (one file, used everywhere):

import { type ClassValue, clsx } from 'clsx';
import { twMerge } from 'tailwind-merge';

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}

// clsx handles conditionals, arrays, objects
// twMerge deduplicates conflicting Tailwind classes

Why you need both:

// Without cn() — conflict! Both p-2 and p-4 get applied.
// Which one wins depends on stylesheet order, not your code.
<div className={`p-2 ${isLarge ? 'p-4' : ''}`}>

// WITH cn() — twMerge resolves the conflict. p-4 always wins:
<div className={cn('p-2', isLarge && 'p-4')}>

// Real-world button component:
function Button({ className, variant = 'primary', ...props }) {
  return (
    <button
      className={cn(
        // Base styles always present
        'inline-flex items-center justify-center rounded-lg font-semibold transition-all',
        // Variant-based styles
        {
          'bg-blue-500 text-white hover:bg-blue-600': variant === 'primary',
          'border border-gray-300 hover:bg-gray-50':   variant === 'outline',
          'text-blue-500 underline-offset-4 hover:underline': variant === 'link',
        },
        // Allow external className overrides
        className
      )}}
      {...props}
    />
  );
}
cn() is everywhere in AI-generated code. If you see cn() in code from ChatGPT, Copilot, or v0.dev, this is what it is. Install clsx + tailwind-merge and create this file. It's the standard pattern across the entire React + Tailwind ecosystem.

Common Mistakes & Mental Models

Most Tailwind frustration comes from a small set of repeated mistakes and incorrect mental models. Fixing these unlocks everything else.

Top 10 Mistakes

01
Using sm: for mobile styles

Everyone does this. sm: means "640px and above", not "small screens". Mobile-first means the unprefixed class IS the mobile style.

Fix: Write mobile styles without prefix. Add sm: / md: to override for larger screens.
02
Building class names dynamically

bg-${color}-500 will never work. Tailwind scans static text — the full class string must exist literally.

Fix: Map complete strings: const map = { red: 'bg-red-500', blue: 'bg-blue-500' }
03
Replacing entire theme keys instead of extending

Putting custom colors in theme.colors (not theme.extend.colors) deletes all built-in colors.

Fix: Always use theme.extend.* unless you explicitly want to replace all defaults.
04
Overusing @apply

@apply recreates the problems Tailwind solves — you're back to managing class names and specificity conflicts.

Fix: Use components (React/Vue components) to reuse styles, not @apply. Reserve @apply for external HTML you can't touch.
05
Forgetting dark: variants in new components

Adding a new bg-white text-black card without dark variants breaks dark mode for that component.

Fix: Develop the habit: every light color gets a dark counterpart. bg-white dark:bg-gray-800.
06
Using pixel values in arbitrary brackets instead of the scale

mt-[16px] when mt-4 exists. Arbitrary values bypass the design system and create inconsistency.

Fix: Always check if the scale has what you need first. Use arbitrary values for genuinely unique one-offs only.
07
Not using gap with flex/grid — using margins instead

gap-4 on the flex/grid container is always cleaner than adding mr-4 to each child.

Fix: Reach for gap-* first on any flex or grid container.
08
Missing overflow-hidden on rounded containers

A rounded-xl card with an image inside — the image corners poke out. Classic gotcha.

Fix: Pair rounded-xl overflow-hidden on any container that clips children.
09
Forgetting min-w-0 in flex children

Flex items have min-width: auto by default — they refuse to shrink below their content width. Causes overflow bugs with long text.

Fix: Add min-w-0 to any flex child that should be allowed to shrink and truncate text.
10
Treating Tailwind like Bootstrap

Looking for .card, .navbar, .btn component classes that don't exist. Tailwind is a utility toolkit, not a component library.

Fix: Build your own components, or use shadcn/ui/Tailwind UI on top. Tailwind gives you the ingredients, not the dish.

The Key Mental Model Shifts

Traditional CSS ThinkingTailwind Thinking
"I'll create a .card class for this""I'll compose utilities directly. If I need to reuse, I'll make a component."
"This class name is too long""The class name IS the documentation. Verbosity is a feature."
"I need to write separate responsive CSS""Every utility has a responsive prefix built in — md:grid-cols-2"
"Dark mode needs a separate stylesheet""Every utility has a dark prefix — dark:bg-gray-800"
"I worry about CSS specificity conflicts""One class = one rule. No specificity wars possible."
"I need to name things consistently""Names are pre-decided by Tailwind's API. No naming decisions."
"The CSS file is getting big""There is no CSS file to maintain."
Adam Wathan on the "mess" objection: "I've had people show me their 'messy' Tailwind HTML. Every single time, it's cleaner than the CSS it would have replaced. The mess isn't in the HTML — it's in trying to maintain a traditional CSS codebase at scale."
